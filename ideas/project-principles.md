# Project Principles

These principles should guide product, design, and technical decisions.

## User Control

Users should gain control by signing up, not lose freedom by staying guests.

Examples:

- claim support history
- choose contribution visibility
- link or unlink accounts
- choose whether a linked account can log in
- trace eligible money/support flow
- delete identifiable account data

## Transparency Without Dark Patterns

The system should be honest about money, sponsors, affiliates, donations, credits, and project changes.

Support should feel informed and voluntary. Important options should be findable without being pushed manipulatively.

## Privacy-first Defaults

Profiles, support history, linked accounts, and personal details should start private or minimal unless the user chooses otherwise.

The site should collect only what is needed for security and features.

## Website as Source of Truth

The website should be the central source for schedules, projects, links, profiles, support history, stream goals, bot commands, blog posts, and overlay state.

Platforms like Twitch, YouTube, Discord, RSS, and socials should sync from the website where possible.

## Manual Override Always Wins

Automation can help, but the streamer should always be able to override it quickly.

This applies to:

- AI assistant behavior
- chat visibility
- overlay events
- sponsor spots
- price changes
- stream goals
- action panel approvals
- schedule updates

## Accessibility and Energy Support

The stream tools should reduce effort, not create new pressure.

The AI assistant should help carry social load when needed. It should not nag, guilt, or demand performance.

## Public Accountability With Boundaries

The streamer may choose to be open about health, personal context, past mistakes, failed businesses, and life circumstances.

The site should support that openness while still allowing careful wording, privacy boundaries, and updates over time.

## Build Small, Test, Improve

The project should move in small loops:

1. build a small piece
2. test it in a realistic setup
3. improve it
4. test again
5. move to the next piece

## Money Must Be Boring

Money, credits, refunds, withdrawals, and donation allocations should be explicit, auditable, and ledger-based.

No silent total edits. No unclear reappropriation. No hidden rules.
