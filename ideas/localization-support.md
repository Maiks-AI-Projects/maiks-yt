# Localization Support

## Idea

Build localization support from the start, even if the website launches in English.

The likely future languages are English and Dutch.

## Why It Matters

Retrofitting localization later can be painful. Adding the structure early makes future translation easier for blog posts, social posts, legal pages, project pages, and personal context pages.

## Data Needed

- locale codes
- translated UI strings
- content language
- fallback locale
- localized slugs if needed
- translation status

## Build Requirements

- localization framework
- English default content
- Dutch-ready structure
- localized date/time/currency formatting
- translation workflow for blog/legal/content pages
- fallback behavior for missing translations

## Open Questions

- Should user-generated content be translated?
- Should blog posts support multiple languages?
- Should legal pages have Dutch versions before launch?
