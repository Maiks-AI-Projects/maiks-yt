# Local Model Lab Workflow

This document describes how to safely experiment with local models in this repository. It is intended for developers who want to generate draft code, documentation, or other artifacts using a local model while keeping the main branch protected.

## 1. Branching Strategy
- Create a new feature branch from `main` (or the appropriate base). Example:
  ```bash
  git checkout -b codex/local-model-lab-<feature>
  ```
- Do **not** commit directly to any protected branches.
- Keep the branch name descriptive and short.

## 2. Generating Drafts with a Local Model
1. Run your local model locally (e.g., `node scripts/run-local-model.js`).
2. Capture the output into files or patches.
3. Commit the draft changes to your feature branch.
4. Verify that all tests pass locally: `pnpm test` for the affected package, and run the architecture checker: `node scripts/check-architecture.mjs`.

## 3. Review Process
- **Local model drafts are not final** – they must be reviewed by a senior developer or the project coordinator.
- Submit a Pull Request (PR) from your feature branch to `main`.
- In the PR description, include:
  - A brief summary of what was generated and why.
  - Any known limitations or risks.

## 4. PR Checklist
| Item | Status |
|------|--------|
| All unit tests pass locally | ✅ |
| Architecture checks (`node scripts/check-architecture.mjs`) passed | ✅ |
| `git diff --check` has no whitespace errors | ✅ |
| Code is reviewed by a senior developer | ⬜ |
| Risks are documented and mitigated | ⬜ |

## 5. Safety Rules
- **Never** push local model changes to production or deploy branches without explicit approval.
- Do not commit secrets or credentials generated by the model.
- If the model generates code that interacts with external services, verify API keys and permissions before merging.
